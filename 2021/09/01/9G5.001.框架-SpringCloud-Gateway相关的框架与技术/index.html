1. springcloud gateway、Spring WebFlux、Reactor关系

  1. springcloud gateway框架，使用了spring weblux框架，而spring webflux框架，底层使用的是Reactor库；
  2. Reactive Streams是规范，Reactor库实现了 Reactive Streams【响应式编程】

WebFlux is based on Reactor, which is a reactive-stream implementation.

2. Spring WebFlux核心组件

  1. Spring WebFlux是异步，非阻塞的。
  2. 它是完全非阻塞的，支持反应流背压(Reactor backpressure)，并在 Netty，Undertow 和 Servlet 3.1+容器等服务器上运行。WebFlux 默认情况下使用 Netty 作为服务器;
  3. Spring WebFlux 是一个异步非阻塞式的 Web 框架，所以，它特别适合应用在 IO 密集型的服务中，比如微服务网关这样的应用中。
  4. WebFlux有Mono，Flux两种集合类型。算是WebFlux的核心组件

Mono

实现Publisher并返回0或1个元素;

Flux

实现Publisher并返回N个元素。

如何更好的理解Mono和Flux

  • Mono / Flux 都是Publisher（发布者），也就是被观察者(Observable)。
  • Mono / Flux 都实现了 org.reactivestreams.Publisher 接口，在 Web Flux，你的方法只需返回 Mono / Flux 即可。你的代码基本也只和 Mono 或 Flux 打交道。而 Web Flux 则会实现 Subscriber ，调用onNext()时将业务开发人员编写的 Mono/Flux 转换为 HTTP Response 返回给客户端。

3.Web Flux(Netty)跟Web MVC(Servlet)优缺点？

  1. Web Flux的效率（qps）比Web MVC高。
  2. Web Flux的编程难度高，维护成本高，对于切面等不好编写。

4.webflux框架中的回压/背压（backpressure）

引自国外技术论坛

Backpressure is a way of dealing with a data stream that may be too large at times to be reliably processed. The goal is to feed the data to subscribers at the rate at which they can reliably deal with that data. The unprocessed data can be buffered (or we could choose different strategy), hence the pressure analogy! Think of a water pressure and a firefighter’s hose as in the featured picture. Firefighter only lets as much water out as she can deal with.

回压是一种处理策略或机制，用于解决：生产者速度大于消费者速度的场景。

  • Reactive API里的提供了5个回压策略：
  • IGNORE: 完全无视的下游背压请求
  • ERROR: 下游跟不上时，抛出IllegalStateException异常信号
  • DROP: 当下游处理不过来时，直接丢弃输入信号
  • LATEST: 下游只会得到上游的最新信号
  • BUFFER: 如果下游无法，缓存所有信号，注意这是一个无界缓冲区，可能会导致OutOfMemoryError

Reactive Stream 里的 API 规定是由订阅者（消费者）调用 request(n) 方法向生产者请求 n 个消息，生产者再调用 onNext() 将 n 个消息提供给消费者。消费者可以按需要获取，生产者也可以按需生产，从而实现背压。
