什么是事务

事务是数据库管理系统（DBMS）执行过程中的一个逻辑单位，由一个有限的数据库操作序列构成。

数据库读一致性会带来三个问题：

  • p1:脏读 一个事务能读取到其他事务未提交的数据
  • p2:不可重复读 一个事务能读取到其他已提交事务（这个事务是Update/Delete）的数据
  • p3:幻读 一个事务能够读取到其他已提交的事务（这个事务是Insert）数据
  • 注意看到上面的不可重复读 和 幻读，他们很类似，都是一个事务能够读取到其他已提交事务的数据，不同点在于其他事务是Insert，还是Update/Delete。
  • 关于不可重复读 和 幻读的区别，网上有很多错误的说法，谷歌查询现实的第一条，与本文的观点是一致，是对的：

  • 如果上面还不足以支撑不可重复读 和 幻读的区别，看看淘宝的数据库内核月报 － 2017 / 06：

  • 以下是sql92标准里给出的不可重复读和幻读的定义：

幻读(phantom read): SQL-transaction T1 reads the set of rows N that satisfy some . SQL-transaction T2 then executes SQL-statements that generate one or more rows that satisfy the used by SQL-transaction T1.If SQL-transaction T1 then repeats the initial read with the same , it obtains a different collection of rows.
上面是sql92标准里给出的幻读的定义，就是在一个事务中的一次查询之后，有另一个事务进行了插入，插入的内容是满足上面查询条件的，如果这个事务当中还有第二次同样查询条件的查询，那么就会导致第二次查询得到的结果集增多。

不可重复读(nonrepeatable read): SQL-transaction T1 reads a row.SQL-transaction T2 then modifies or deletes that row and perfom a COMMIT. if T1 then attempts to reread the row, it may receive the modified value or discover that the row has been deleted.
这是不可重复读里在SQL92标准里面的定义，可以看出是针对一条记录的；并且事务T2的操作是 modifies or deletes

四个隔离级别和三个读一致性问题的解决关系列表：

  • RU 有p1, p2, p3 三个问题
  • RC 解决p1 会有p2, p3 两个问题
  • RR 解决p1, p2 会有p3 一个问题，但是MySQL的InnoDB存储引擎可以通过MVCC解决幻读
  • Serializable 解决p1, p2, p3

  • 以上的四个隔离级别和和三个读一致性解决关系，也可以看http://www.contrib.andrew.cmu.edu/~shadow/sql/sql1992.txt

如何保证一个事务中前后两次（无论多少次）读取数据结果一致，实现事务隔离，应该怎么做?

  • 答案是使用：当前读 和 快照读
  • 当前读–在读取数据前，对其加锁，阻止其他事务对数据进行修改-Lock Based Concurrency Control(LBCC)
  • 快照读–生成一个数据请求时间点的一致性数据快照，并用这个快照来提供一定级别（语句级或事务级）的一致性读取-Multi Version Concurrency Control(MVCC)

几个FAQ问题

mysql到底是锁住了什么，来实现并发资源的控制？

其实不是锁行，也不是锁表，本质锁的是索引
锁都是针对索引创建的, 而不是记录本身, 如果没有索引, 就会使用隐藏的主键索引

查询的列没有设计成索引或者没有用到索引的时候，为什么会锁表

表中没有显示定义主键索引，或是任何索引，会自动创建一个隐藏索引：6个字节的ROW_ID，作为聚簇索引，会锁住这个聚簇索引树，也就是锁表。

为什么使用name（name是唯一索引）加锁，会加锁阻塞id（id是主键）的条件查询

原因是 name是二级索引/辅助索引，叶子结点存放的是id主键值，因此会锁住id条件的查询

MVCC 在 RC 和 RR 两种隔离级别下都有，在RC级别下，MVCC为什么解决不了幻读?

MVCC在这RC和RR下，两者建立快照的时间点不一样
RC的快照建立时间是当前select，所以能够看到其他事务已提交的数据；
RR的快照建立时间是第一次查询的时候，所以未提交的事务和其后的事务的数据是不可见的；

MySQL在隔离级别RR下，到底能否避免幻读？

  • 其实，用下面准确的概念和术语：

    按照SQL92的规范，隔离级别RR下，是避免了脏读和不可重复读，但是不能避免幻读；但是MySQL在Innodb存储引擎下，为了避免在RR级别下的幻读问题，使用了MVCC和临建锁（Next-key Lock）的锁算法，来避免或者说解决幻读的问题。

MySQL在InnoDB存储引擎下，在隔离级别RR下，什么时候查询使用了MVCC，什么时候查询用了Next-key Lock？

  • 在InnoDB存储引擎下，当查询属于快照读 （select… from …）的时候，使用MVCC算法来避免幻读；
  • 在InnoDB存储引擎下，当查询属于当前读(select…from…for update/seleect…from… lock in share mode/insert/update/delete)的时候，使用临建锁（Next-key Lock）算法来避免幻读；
  • 临建锁 Next-key Lock 只在RR隔离级别下面才有；MVCC在RC和RR 2个隔离级别下都有；

隔离级别是RR通过什么手段解决了不可重复读？

  • 是通过MVCC或临建锁解决的，具体什么时机用MVCC或临建锁，参考上面的问题答案
