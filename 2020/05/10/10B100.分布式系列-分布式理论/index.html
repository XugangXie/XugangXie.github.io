分布式系统

什么是分布式系统，用通俗的话来说：分布式系统是多个节点连同后的系统
个人认为，分布式，是个广义的概念，可以体现在具体的中间件内部架构上、应用服务的落地实现等。

分布式系统中会遇到的问题

分布式系统中会遇到很多问题，其中一个很重要的是：数据一致性 和 容错性

数据一致性分为：以数据为中心的一致性模型 和 以客户为中心的一致性模型

数据一致性

以数据为中心的一致性模型

一致性模型实际上是进程和数据存储之间的一个约定。
理想情况下，在任何一个副本上执行的读操作，应当返回的是该数据在其最后一次写操作之后的结果。
在没有全局时钟的情况下，精确定义哪次写操作是最后一次是十分困难的。于是有以下几种粒度的一致性模型：

  1. 严格一致性（Strick Consistency）
    任何的读操作都要读到最新的写结果，这是限制最强的模型。依赖于绝对的全局时钟。在分布式系统中实现该模式代价太大，所以在实际中运用有限，基本上不可能做到。

  2. 持续性一致性（Continuous Consistency）
    有多种不同的策略，来为应用程序指定他们能容忍哪些不一致性，策略有：
    副本之间数据的偏差
    副本之间新旧程度偏差
    更新操作顺序的偏差

  3. 顺序一致性（Sequential Consistency）

  4. 因果一致性（Casual Consistency）

  5. 入口一致性（Entry Consistency）

以客户为中心的一致性模型

以客户为中心的一致性也就是从用户视角来看，数据是一致的，客户只关心数据最终是否会一致。
以客户为中心的一致性模型分为以下几类：

  1. 单调读一致性
  2. 单调写一致性
  3. 读写一致性
几个中间件的数据一致性实现
  1. Consul使用Raft算法实现集群数据一致性；
  2. Zookeeper
  3. Eureka
  4. Redis
  5. Kafka

容错性

分布式系统要有容错性，集中式系统中任何一个组件故障，都会导致整个系统无法使用，而分布式系统中，它容许部分失效，但整个系统应该在恢复的同时可以继续提供服务。
也就是说，他应该能容忍错误，在发生错误时，某种程度上可以继续操作。

CAP理论和BASE理论

CAP理论：

  • C：数据一致性
  • A：服务可用性
  • P：网络分区容错性

开发大规模的分布式系统时会遇到三个特性：一致性（Consistency）、可用性（Availability）、分区容错（Partition-tolerance），而一个分布式系统最多只能满足其中的两项。
基于CAP理论，一个分布式系统，要么满足CP，要么满足AP。

任何一个分布式架构都必须满足网络分区的容错性，所以系统架构师往往需把精力放在如何根据业务特点，在C和A之间寻求平衡。

比如Zookeeper集群架构，满足CP，在分布式集群环境下，内部用的是Paxos算法，若有单点故障，同时在网络出现问题时，会影响ZK的选举造成不可用。

SpringCloud Eureka里的Eureka，满足的是AP，它保证里注册中心的高可用性，但舍弃了数据的强一致性，但会最终一致性。

而前面我们提到的JTA 两阶段提交协议的分布式事务方案，强调的就是一致性；由于可用性较低，实际应用的并不多。

BASE理论

BASE理论是基于CAP理论演化而来的，核心思想是：即便不能达到强一致性，但也可以根据应用的特点，采用适当的方式来达到最终一致性。

  • 基本可用（Basically Available）
    指分布式系统在出现不可预知故障的时候，允许损失部分可用性。

  • 软状态（ Soft State）
    指允许系统中的数据存在中间状态，并认为该中间状态的存在不会影响系统的整体可用性，即允许系统在不同节点的数据副本之间进行数据同步的过程存在延时。

  • 最终一致（ Eventual Consistency）
    强调的是所有的数据更新操作，在经过一段时间的同步之后，最终都能够达到一个一致的状态。因此，最终一致性的本质是需要系统保证最终数据能够达到一致，而不需要实时保证系统数据的强一致性。

分布式系统中，服务的调用方式

分布式系统中，服务的调用方式可以有：

  1. （面向过程的调用）同步的方式，以RPC形式，进行远端服务的调用；
  2. （面向消息的通信）PRC的调用是同步的特性，有时有的场景是需要异步的，那么此时就可以引入消息中间件；

几个中间件在分布式这块的体现

  1. Zookeeper
    Zookeeper的leader选举，本质用的是Paxos算法，内部使用的是ZAB广播协议，它是一种基于消息传递且具有高度容错特性的一致性算法。
    Paxos算法的具体实现，是fast-election

  2. Eureka

  3. redis
    gossip协议
    数据以slot形式分片

  4. kafka

分布式系统架构体系

可以分为：面向服务的SOA架构、微服务架构MSA、无服务架构Serverless

面向服务的SOA架构

SOA不是一项技术，也不是一个标准，而是一种架构
《Enterprise SOA》中对SOA的定义：

SOA是一种软件架构，它包含了4个关键概念：应用程序前端、服务、服务库和服务总线。一个服务包含合约、一个或者多个接口以及一个实现。

SOA架构的框架，比如JBoss ESB
WebService也是SOA架构的一个应用

SOA架构的演变：Rest风格的架构，微服务架构，Serverless架构
RESTful架构：提供基于HTTP协议的接口，接口都是无状态的。

  1. 所有的接口可以认为是资源，对资源的操作就涉及到增删改查，那么在RESTful上的体现，则是通过Http Method的几个参数来体现：get post put delete；
  2. 使用Http code来表示错误的处理；

微服务架构MSA

将业务功能服务化是SOA架构的延续，RESTful架构则是可以以更加轻量级的协议进行通讯；DDDD领域驱动设计指导我们如何分析并模型化复杂业务；CI/CD 持续集成/交付，促使我们构建更快、更可靠的软件部署和交付能力；虚拟化和容器的发展，使我们简化了部署环境的创建、安装；DevOps文化的流行以及全栈自治团队的出现，使得小团队更加全功能化，这些都是推动微服务架构诞生和发展的重要因素。

MSA本身没有严格的定义，它也是一种架构风格，有以下几个 特点：

  1. 每个微服务可以独立运行，独立部署；
  2. 每个微服务之间通过一些轻量级的通讯机制进行通讯，比如Rest；
  3. 每个微服务内部可使用不同的技术实现，可有自己独立的数据库，分散数据管理；
  4. 各个微服务之间是松耦合的,分散治理；
  5. 基础设施自动化，微服务团队更加依赖于基础设施的自动化；

由于是微服务架构，每个微服务都有自己的数据库，因此数据库的垂直拆分，天然在MSA中得到体现。
垂直拆分：把数据库中的不同业务数据，拆分到不同的数据库中。

MSA中最核心的是服务治理，而服务治理最基础的组件是注册中心
服务治理的解决方案：Dubbo和SpringCloud Eureka、Connsul
注册中心的方案有：Etcd、Consul、Zookeeper
